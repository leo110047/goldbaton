# Phase 1 risk spikes

This phase validates the two provider control surfaces before the Provider
interface is designed. The code in `spikes/` is an executable probe, not a
production provider implementation.

## Required evidence

| Provider | Streaming | Approval round trip | Interrupt | Protocol types |
|---|---|---|---|---|
| Codex app-server | `item/agentMessage/delta` | `item/commandExecution/requestApproval` followed by a typed `decline` response | covered by app-server protocol, exercised in Phase 2 | generated by the installed CLI and compiled with TypeScript |
| Claude Agent SDK | partial `text_delta` | `canUseTool` callback followed by an `allow` result | `Query.interrupt()` resolves | official SDK package types |

Every live probe exits non-zero if required evidence is missing. The prompts and
tool inputs are canonical English. The Codex probe declines the requested
command. The Claude probe allows only a 30-second Node timer and interrupts it
after 500 ms.

## Commands

```bash
npm run check
npm run spike:codex
npm run spike:claude

# Runtime comparison
npm run spike:all:bun
```

`spike:generate:codex` rebuilds `spikes/codex/generated/` from the installed
Codex CLI. Generated bindings are intentionally not committed because they are
version-specific and reproducible from the CLI under test.

## Environment contract

- `codex` must be installed, authenticated, and expose `app-server generate-ts`.
- Claude Agent SDK `0.3.206` bundles Claude Code `2.1.206` and requires Node 18
  or newer.
- A live Claude run needs an Anthropic-supported authentication method. Product
  use cannot assume that a third-party application may reuse a user's
  `claude.ai` login.
- Live probes consume provider tokens. Provider turns do not mutate tracked
  project files; the Codex command does regenerate ignored protocol bindings
  under `spikes/codex/generated/` before the turn.

## Windows rerun

Use a real Windows machine so Codex and Claude can use their normal interactive
login state. This manual route does not require repository API-key secrets.
After confirming both CLIs can start an authenticated turn, run from PowerShell:

```powershell
node --version
bun --version
codex --version
claude --version
npm ci
npm run spike:verify
```

The command must finish with all repository checks passing and four provider
JSON summaries: Codex and Claude on Node, then Codex and Claude on Bun. Codex
must report `approvalReplies: 1`, `approvalResolved: true`, streamed text, and a
completed turn without creating its marker. Claude must report every evidence
boolean as `true`; `resultSubtype: "error_during_execution"` is expected after
the deliberate interrupt.

As of 2026-07-10, this native live run has not been performed, so Windows
support remains unverified.

Static readiness was checked separately without executing Windows code:

- The forced `win32-x64` install for Claude Agent SDK 0.3.206 contains
  `claude.exe`.
- The Codex 0.144.1 `win32-x64` package contains `codex.exe`, its command
  runner, and Windows sandbox setup binaries.
- Unit tests verify that the harness invokes the npm `codex.cmd` shim through
  `cmd.exe` instead of trying to spawn a `.cmd` file directly.
- The credential-free `windows-readiness` CI job runs binding generation,
  repository checks, tests, and the build on `windows-latest`.

These checks prove package availability and launcher construction only. They do
not prove authentication, streaming, approval, interrupt, or sandbox behavior
on Windows.

If the native run fails for a platform reason, save the failing command and full
error, then run the same `npm ci` and `npm run spike:verify` sequence in WSL.
Windows support is not decided until the native result, or native failure plus
WSL result, has current evidence.

## Result record

The authoritative result is fresh command output, not this document. The
2026-07-10 macOS arm64 run produced this matrix:

| Runtime | Codex 0.144.1 | Claude SDK 0.3.206 / Claude Code 2.1.206 |
|---|---|---|
| Node 24.16.0 | pass: text delta, approval request, declined response, completed turn | pass: text delta, `canUseTool` response, interrupt resolved |
| Bun 1.3.11 | pass: same evidence | pass: same evidence |

The Claude terminal result is `error_during_execution`, which is expected after
the deliberate interrupt. Codex generated 671 TypeScript files, and TypeScript
7.0.2 compiled the generated protocol plus both harnesses without errors.

Node 24 is the production baseline because it is the SDK's declared runtime
contract and the lowest-risk Windows/CI path. Bun remains a tested compatibility
target and can be reconsidered for Phase 8 single-binary packaging.
